# MENZO QR v2.0 — Этап 3: Frontend (база)

Next.js 15 (App Router) + TypeScript + Tailwind + shadcn-style компоненты на Radix.

## Установка в вашей среде

```bash
npx create-next-app@latest menzo-qr-frontend --typescript --tailwind --app --src-dir --import-alias "@/*"
cd menzo-qr-frontend
# скопируйте файлы этого архива поверх созданного проекта
npm install
cp .env.example .env.local
npm run dev
```

Убедитесь, что `NEXT_PUBLIC_API_URL` в `.env.local` указывает на ваш backend (`http://localhost:8000/api/v1` по умолчанию).

## Дизайн-система этого этапа

- **Палитра:** `background` #F8FAFC, `surface` (карточки) #FFFFFF, `ink` (текст) #0F172A, `muted` (второстепенный текст) #64748B, `border` #E2E8F0, `accent` #2563EB, `success` #10B981, `warning` #F59E0B, `danger` #EF4444. Универсальная светлая premium-палитра без тематической привязки к конкретному типу заведения — подходит кофейне, ресторану, бару или фудкорту одинаково.
- **Типографика:** единое семейство Inter (для заголовков и интерфейса) + JetBrains Mono (номера заказов, коды, технические данные) — подключены через `next/font/google` в `app/layout.tsx`. Чистый гротеск в духе Linear/Stripe/Vercel, без декоративной засечки.
- **Радиусы:** 20px (`rounded-card`) и 24px (`rounded-card-lg`) — крупные белые карточки без острых углов.
- **Glass/Blur:** `.glass-surface` и `.glass-surface-strong` в `globals.css` — полупрозрачные белые карточки с `backdrop-blur` поверх мягкого голубовато-зелёного градиента фона.
- **Skeleton/Shimmer:** класс `.skeleton` — используйте вместо статичных плейсхолдеров на всех экранах с загрузкой данных (Этапы 4–5).
- **Фирменная деталь:** класс `.scan-frame` — уголки видоискателя QR-сканера на ключевых карточках (QR-коды, активный заказ), плюс `.status-pulse` — живой пульсирующий индикатор для статусов "новый заказ"/"Telegram подключен". Обе детали отсылают к сути продукта (сканирование QR), а не к конкретной нише кофеен.

## Что реализовано

- Полная конфигурация проекта (Tailwind-токены, шрифты, PostCSS, TS-алиасы).
- `AuthProvider` (React Query + localStorage-токен) и `ToastProvider` (Radix Toast).
- API-клиент (`lib/api/client.ts`) с автоматической подстановкой Bearer-токена и обработкой 401.
- Карта эндпоинтов backend (`lib/api/endpoints.ts`), синхронизированная с маршрутами Laravel из Этапа 2.
- Библиотека UI-примитивов: `Button`, `Card`, `Input`, `Label`, `Badge`, `Skeleton`, `Toast` — все стилизованы согласно дизайн-системе.
- Полностью рабочие страницы **входа** и **регистрации** с валидацией Zod + React Hook Form и реальными вызовами API.
- Каркас панели владельца (`(owner)/layout.tsx`): защита маршрутов (редирект на `/login` для неавторизованных), верхняя панель с меню пользователя и выходом.

## Этап 4: Функционал владельца (добавлено в этом архиве)

- **`/cafes`** — список кафе владельца, создание нового кафе (диалог).
- **`/cafes/[cafeId]/menu`** — категории и блюда: создание/редактирование/удаление категорий, блюд с фото, вариантами (размер/объём) и дополнениями (динамические поля через `useFieldArray`).
- **`/cafes/[cafeId]/qr-codes`** — создание QR-кодов для стола или самовывоза, скачивание изображения, удаление. Карточка QR использует фирменную деталь `.scan-frame`.
- **`/cafes/[cafeId]/orders`** — канбан-доска заказов по статусам (Новый → Принят → Готовится → Готов → Выдан), кнопка перехода на следующий статус, отмена заказа, автообновление каждые 15 секунд. Новые заказы отмечены `.status-pulse`.
- **`/cafes/[cafeId]/staff`** — добавление сотрудников (менеджер/персонал), удаление доступа.
- **`/cafes/[cafeId]/telegram`** — получение кода привязки, статус подключения (не придумываем username бота — он подставляется из `NEXT_PUBLIC_TELEGRAM_BOT_USERNAME`, который вы заполните на Этапе 6).
- **`/cafes/[cafeId]/settings`** — название, адрес, телефон, валюта, логотип.
- Общий каркас: `CafeSidebar` с навигацией между разделами конкретного кафе.

## Этап 5: Публичное меню (добавлено в этом архиве)

Маршрут **`/menu/[cafeSlug]`** — публичная страница, которую видят гости по QR-коду. Отдельная группа маршрутов `(public)`, без панели владельца.

### Реализовано

- **Sticky-хедер** с логотипом и названием кафе + **sticky-навигация по категориям** с горизонтальным скроллом, скролл-спаем (`IntersectionObserver`) и плавным «плавающим» индикатором активной категории (`layoutId` в Framer Motion — shared element transition).
- **Карточки блюд** — премиальный вид: фото, бейджи (Новинка/Хит/Острое/Халяль/Вегетарианское/Нет в наличии — все реальные, управляются владельцем в форме блюда на Этапе 4, не выдуманы), цена, плавное появление с лёгким stagger-эффектом.
- **Bottom Sheet блюда** — фото, варианты, дополнения, комментарий, степпер количества, живой пересчёт суммы, кнопка добавления. Реализован на Radix Dialog + Framer Motion: spring-анимация выезда снизу и drag-to-dismiss с упругим сопротивлением — ощущается как нативный iOS-шит.
- **Корзина** — тот же Bottom Sheet-примитив, мгновенное изменение количества, пересчёт суммы без перезагрузки, плавающий бар корзины с spring-анимацией появления.
- **Оформление заказа** — минимум полей, ровно как в ТЗ: самовывоз (имя → телефон → комментарий), заказ в заведении (имя → комментарий, без телефона — стол уже известен персоналу).
- **Отслеживание заказа** (`/menu/[cafeSlug]/order/[orderNumber]`) — визуальный степпер статуса с автообновлением, бонус сверх ТЗ, использующий уже готовый публичный API статуса заказа.
- **Производительность** — `next/image` с `lazy` для карточек и `sizes` для адаптивной загрузки, React Query кэш (`staleTime`), skeleton-заглушка на все состояния загрузки.
- **Accessibility** — все интерактивные элементы (степпер, варианты, допы) не менее 44×44px, фокус-стили, `aria-label` на иконках-кнопках.

### Изменения в backend, необходимые для этого этапа

- **Бейджи блюд не выдуманы** — добавлены реальные поля `dishes.tags` (json) и `dishes.is_available` (миграция, модель, DTO, ресурс, валидация). Управляются владельцем в форме блюда (Этап 4, тоже обновлён — переключатель наличия и чипы тегов).
- **Заказ в заведении без телефона** — `orders.customer_phone` теперь nullable в БД; `customer_phone` обязателен только для самовывоза (`required_if:type,takeaway`) — соответствует минимальному сценарию чек-аута из ТЗ, без придумывания данных.
- **QR-ссылка для стола** теперь передаёт `table_id` и `table_number` в query-параметрах (`?table_id=5&table_number=12`), так как API заказа требует числовой `table_id`, а не номер стола.

Все три группы изменений уже включены в актуальный `menzo-qr-stage2-backend.zip`.
